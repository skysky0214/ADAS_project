## 목적

이 문서는 2026-05-11 `gh` workspace에서 진행한 ADAS 통합 파이프라인 변경사항을 정리한다.

최종 목표는 실차 또는 rosbag 기반 LiDAR 입력을 받아 아래 흐름을 실시간으로 연결하는 것이다.

```text
LiDAR PointCloud2
-> DSVT perception
-> pedestrian filtering
-> pedestrian tracking
-> SR-LSTM trajectory prediction
-> planner-facing snapshot
-> TTC warning candidate
```

## 현재 통합 흐름

현재 확인된 흐름은 다음과 같다.

```text
ros2 bag play
-> /lidar_points
-> app/main.py
-> OpenPCDet DSVT adapter
-> PedestrianTracker
-> SR-LSTM adapter
-> PlannerSceneSnapshot
-> TTC warning adapter
-> artifacts 저장
```

`rosbag2_2026_04_02-17_34_49`의 `/lidar_points` topic으로 실제 publish/subscribe 환경을 재현했다.

## 트래킹 연결

인지 결과 중 `Pedestrian`만 필터링해 `PedestrianTracker`로 넘긴다.

현재 tracking 입력:

```text
PedestrianDetection(score, x, y, z, dx, dy, dz, heading)
```

현재 tracking 출력:

```text
TrackedPedestrian(track_id, x, y, z, vx, vy, missed, score, dx, dy, dz, heading, history)
```

트래킹 품질은 아직 좋지 않을 수 있다. 현재 작업 범위에서는 perception 결과를 tracking stage까지 전달하고 ID/history를 생성하는 연결을 우선 확인했다.

## 예측 모델 연결

SR-LSTM realtime predictor를 현재 tracking 결과에 연결했다.

추가 파일:

- `app/prediction/adapters/srlstm_predictor.py`

SR-LSTM adapter는 `TrackedPedestrian` 목록에서 아래 형태를 만들어 내부 predictor에 넘긴다.

```python
{
    track_id: (x, y)
}
```

SR-LSTM은 관측 4프레임이 쌓인 뒤부터 미래 8개 point를 예측한다.

출력 타입:

```text
PredictedTrajectory(track_id, points, confidence, model_name)
TrajectoryPoint(t_sec, x, y)
```

검증 결과:

```text
frame=0 ... predicted=0
frame=1 ... predicted=0
frame=2 ... predicted=0
frame=3 ... predicted=25
```

4번째 frame부터 예측이 생성되는 것은 SR-LSTM의 `obs_length=4` 조건 때문에 정상이다.

## Planner Bridge 정리

기존 bridge는 예전 타입인 `TrackedObject`, `PlannerObjectView`, `PlannerSceneSnapshot`을 기대하고 있어 import가 깨져 있었다.

수정 내용:

- `PlannerObjectView` 추가
- `PlannerSceneSnapshot` 추가
- `build_planner_snapshot()` 입력을 `TrackedPedestrian` 기준으로 변경

현재 bridge 흐름:

```text
list[TrackedPedestrian] + list[PredictedTrajectory]
-> PlannerSceneSnapshot
```

planner object에 담는 값:

- `track_id`
- `label="Pedestrian"`
- `current_position=(x, y, z)`
- `current_velocity=(vx, vy)`
- `predicted_path=[(x, y), ...]`
- `missed`

runtime node는 매 frame마다 `planner_snapshots.json`도 저장한다.

## TTC Warning Adapter

하드웨어/제어팀과 실제 제어 interface를 확정하기 전까지는 차량 command를 직접 만들지 않고, 경고 단계와 감속도 후보만 산출한다.

추가 파일:

- `app/control/ttc_warning.py`

입력:

```text
list[TrackedPedestrian] + list[PredictedTrajectory]
```

출력:

```text
TTCWarning(frame_id, timestamp_sec, track_id, level, label, action,
           min_ttc_sec, target_accel_mps2, distance_m, collision_time_sec)
```

보고서 기준 threshold:

- Level 1 FCW: `TTC <= 1.70s`
- Level 2 partial brake candidate: `TTC <= 1.50s`
- Level 3 AEB candidate: `TTC <= 0.80s`

현재 자차 경로 가정:

- LiDAR/body frame에서 자차는 원점 `(0, 0)`에서 `+x` 방향 직진
- 기본 ego speed는 `10.0 m/s`
- 기본 safety radius는 `1.0 m`

runtime node는 매 frame마다 TTC warning을 계산해 `ttc_warnings.json`으로 저장한다. RViz2 marker에도 warning이 있는 track 위에 `L1/L2/L3`, TTC, 감속도 후보를 표시한다.

기존 전달 파일 `/home/gh/Downloads/kakao_talk/ttc_warning_system.py`의 버그성 부분도 adapter에서 정리했다.

- pandas/CSV 후처리 방식 제거
- live pipeline의 `TrackedPedestrian`와 `PredictedTrajectory` 타입을 직접 사용
- `direct_ttc=0.0`이 `inf`로 바뀌는 조건 처리 문제 제거

## RViz2 시각화

runtime node는 RViz2에서 바로 볼 수 있도록 `visualization_msgs/msg/MarkerArray`를 publish한다.

기본 marker topic:

```text
/adas/tracking_markers
```

marker 구성:

- 파란 box: DSVT detection에서 이어진 tracked pedestrian 3D box
- 흰 text: tracker가 부여한 track ID
- 초록 line strip: 해당 ID가 지나온 history
- 빨간 line strip: SR-LSTM이 만든 예상 경로
- 노랑/주황/빨강 text: TTC warning level과 감속도 후보

RViz2 설정:

```text
Fixed Frame: hesai_lidar
Display 1: PointCloud2 /lidar_points
Display 2: MarkerArray /adas/tracking_markers
```

`rosbag2_2026_04_02-17_34_49`의 LiDAR frame id는 `hesai_lidar`다. 다른 bag에서 frame id가 다르면 node 실행 시 `--marker-frame <frame_id>`로 marker frame을 고정한다.

RViz2 publish가 필요 없는 실행에서는 `--no-rviz`를 붙이면 된다.

## 주요 실행 명령

터미널 1에서 ADAS node 실행:

```bash
cd /home/gh/workspaces/design_project/ADAS_project

/home/gh/anaconda3/envs/pedestrian_gt/bin/python app/main.py \
  --topic /lidar_points \
  --max-frames 20 \
```

## 생성 산출물

runtime node는 output directory 아래에 다음 파일을 저장한다.

```text
detections.csv
tracking_results.csv
tracking_results.json
prediction_input.json
predicted_trajectories.csv
predicted_trajectories.json
ttc_warnings.json
planner_snapshots.json
```

각 파일 의미:

- `detections.csv`
  - DSVT가 낸 raw detection box
- `tracking_results.csv/json`
  - tracker가 부여한 ID와 history
- `prediction_input.json`
  - SR-LSTM 이전 단계의 ID별 observed sequence
- `predicted_trajectories.csv/json`
  - SR-LSTM 미래 궤적 예측
- `ttc_warnings.json`
  - 예측 궤적 기반 TTC warning level과 감속도 후보
- `planner_snapshots.json`
  - planner/control 단계에서 읽기 쉬운 frame별 object snapshot
